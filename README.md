# PTMK Test Task — Система учёта заявок

## Описание проекта

REST API для управления заявками сотрудников.

Проект реализует бизнес-процесс создания, обработки и контроля выполнения заявок:

* создание заявки сотрудником;
* просмотр списка заявок;
* фильтрация заявок по различным параметрам;
* изменение статуса заявки с учётом бизнес-правил;
* формирование отчётной информации по заявкам.

Проект разработан на Python с использованием FastAPI, PostgreSQL и SQLAlchemy 2.0.

# Запуск проекта

## Переменные окружения

Перед запуском необходимо создать файл `.env`.

Пример:

```env
DATABASE_URL=postgresql+asyncpg://postgres:postgres@postgres:5432/ptmk
```

## Docker Compose (рекомендуемый способ)

Проект полностью запускается через Docker Compose.

Требуется:

- Docker Desktop
- Docker Compose

Клонирование:

```bash
git clone <repository_url>

cd ptmk_test_task
```
### Запуск:
```bash
docker compose up --build
```

При запуске автоматически:

```
создаётся контейнер PostgreSQL;
выполняются миграции Alembic;
запускается FastAPI приложение.
```

После успешного запуска:

Проверка состояния контейнеров:

```bash
docker compose ps
```
Ожидаемый результат:

```
ptmk_postgres   healthy
ptmk_api        running
```

API:

http://127.0.0.1:8000

Swagger:

http://127.0.0.1:8000/docs

PostgreSQL автоматически запускается в отдельном контейнере.

Остановка:

```bash
docker compose down
```

Удаление базы вместе с данными:

```bash
docker compose down -v
```

## Генерация тестовых данных с помощью скриптов

Все вспомогательные скрипты запускаются внутри контейнера API.

Общий формат:

```bash
docker compose exec api python -m app.scripts.<script_name>
```

Очистка базы данных

```bash
docker compose exec api python -m app.scripts.clear_database
```

Удаляет тестовые данные из базы.

### Генерация тестовых данных

Настройки количества записей находятся в:

app/scripts/generate_data.py

Пример:

EMPLOYEES_COUNT = 1000
REQUESTS_COUNT = 1000000

Запуск генерации:

```bash
docker compose exec api python -m app.scripts.generate_data
```
Пример результата:

```
Inserted 1000000/1000000 requests

Generation completed successfully.
```

# API Endpoints

## Requests

### Создание заявки

```
POST /requests
```

Создаёт новую заявку.

---

### Получение списка заявок

```
GET /requests
```

Поддерживаемые параметры:

| Параметр    | Описание                     |
| ----------- | ---------------------------- |
| status      | фильтр по статусу            |
| executor_id | фильтр по исполнителю        |
| department  | фильтр по отделу исполнителя |
| overdue     | только просроченные заявки   |
| limit       | количество записей           |
| offset      | смещение                     |

Пример:

```
GET /requests?status=Новая&limit=20&offset=0
```

---

### Изменение статуса заявки

```
PATCH /requests/{request_id}/status
```

Изменяет статус заявки согласно бизнес-правилам.

---
## Employees

### Получение списка сотрудников
```
GET /employees
```

Возвращает список сотрудников с пагинацией.

## Reports

### Получение отчёта

```
GET /reports
```

Возвращает:

* общее количество заявок;
* количество сотрудников;
* распределение заявок по статусам.

Пример ответа:

```json
{
  "total_requests": 1000000,
  "total_employees": 1000,
  "requests_by_status": {
    "new": 333072,
    "in_progress": 333588,
    "completed": 333340
  }
}
```

# Объектная модель

Основные сущности:

## Employee

Сотрудник системы.

Поля:

* id
* ФИО
* подразделение
* должность

---

## Request

Заявка.

Поля:

* id
* номер заявки
* дата создания
* описание
* срок выполнения
* статус
* автор
* исполнитель

Связи:

* одна заявка имеет одного автора;
* одна заявка имеет одного исполнителя;
* сотрудник может иметь множество заявок.

# Бизнес-правила

## Статусы заявки

Используются следующие состояния:

* Новая
* В работе
* Выполнена

---

## Допустимые переходы статусов

Разрешённые переходы:

```
Новая → В работе

В работе → Выполнена
```

Недопустимые переходы отклоняются API.

Например:

```
Новая → Выполнена
```

вернёт ошибку:

```json
{
  "detail": "Invalid status transition"
}
```

## Проверка пагинации

```bash
docker compose exec api python -m app.scripts.show_data \
--table requests \
--limit 20 \
--offset 900000
```

Результат:

* проверена работа пагинации на таблице размером 1 000 000 записей;
* запросы с большим offset выполняются корректно;
* время выполнения зависит от параметров окружения PostgreSQL.

## Реализованные оптимизации

Использованы:

### SQLAlchemy eager loading

Для исключения проблемы N+1 запросов:

```python
joinedload(Request.author)
joinedload(Request.executor)
```

---

### Ограничение количества возвращаемых данных

API использует пагинацию:

```
limit
offset
```

что позволяет работать с большим количеством записей без загрузки всей таблицы.
